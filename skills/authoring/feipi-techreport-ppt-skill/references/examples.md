# 示例

## 示例 1：架构图页

### 用户输入

```
我们做了一个本地多 agent 调度架构。
包含以下组件：
- Codex：负责代码生成和审查
- Claude Code：负责交互式开发辅助
- Qoder：负责需求分析和任务分解
- LiteLLM：作为模型路由层，统一调度不同模型
- 任务队列：Redis 实现的异步任务分发
- 审查器：安全策略检查，拦截不合规请求

数据流：用户请求 → Qoder 分解 → 任务队列 → LiteLLM 路由 → Codex/Claude Code 执行 → 审查器 → 返回结果

核心瓶颈：LiteLLM 路由层的延迟占整体响应时间的 40%。
```

### 信息充足性判断

- 主题：本地多 agent 调度架构 ✓
- 结论：LiteLLM 路由延迟是核心瓶颈 ✓
- 原始事实：6 个组件及其职责、完整数据流 ✓
- 关系：顺序流程 + 路由分发 ✓

四项齐全，信息充足。

### Page Contract 示例

```

- 不补充任何 TPS、延迟毫秒数、模型列表等用户未提供的数据。

## 示例 2：对比矩阵页

### 用户输入

```
对比三种公链建设路线：

1. 串行共识路线
- 代表：传统 EVM 链
- 优点：生态成熟、工具链完善
- 缺点：TPS 瓶颈明显，扩容靠 L2
- 适合：对性能要求不高、重视生态的项目

2. 并行共识路线
- 代表：Solana 类链
- 优点：原生高 TPS
- 缺点：开发门槛高、生态相对封闭
- 适合：高吞吐场景、自建技术团队

3. Multi-proposer 路线
- 代表：新兴多提案者架构
- 优点：去中心化程度高
- 缺点：共识复杂度高、目前缺乏大规模验证
- 适合：重视去中心化、愿意承担技术风险的项目
```

### 信息充足性判断

- 主题：三种公链建设路线对比 ✓
- 结论：各路线适合不同场景，无绝对优劣 ✓
- 原始事实：3 个对象 × 4 个维度（代表、优点、缺点、适合场景）✓
- 关系：并列对比 ✓

四项齐全，信息充足。

### Page Contract 示例

```
## Page Contract

### 3. 使用的原始信息
- 三条路线：串行共识、并行共识、Multi-proposer

### 4. 页面内容范围
本页包含：
- 3 列对比表格（路线 × 维度）
- 顶部推荐倾向标注

本页不包含：
- 具体 TPS 数值（用户未提供，不自造）
- 代码行数 / 开发周期估算（用户未提供）
- 市场判断或投资建议

### 5. 推荐页面结构
- 主表：comparison-matrix 版式，3 个对象 × 4 个维度（代表、优势、风险、适合场景）
- 顶部 KPI/标签区：用 3 个短标签呈现生态成熟、吞吐、去中心化三个取舍轴
- 结论区：Bottom bar 写"路线选择取决于项目优先级，不存在单一最优"

### 6. 版面蓝图
- 主视觉区域：中部 55% 放 3 × 4 重点矩阵，避免长句
- 证据/说明区域：顶部 20% 放三个取舍标签，帮助快速扫描
- 脚注/来源区域：本页不需要脚注，避免占用主体空间
- 需要压缩或不进入主画面的信息：优缺点改写为短标签，不补充 TPS 等未提供数据

### 7. 生成前确认
请确认是否按这个内容范围生成当前页 PPT。
```

### 生成策略说明

- 版式选择 `comparison-matrix`，因为内容是多方案对比。
- 用户没有提供 TPS、LOC、开发周期等具体数值，**绝对不自造**。
- 表格维度使用用户提供的四个维度：代表、优点、缺点、适合场景。

```

## 示例 4：过载对比页降维

### 用户输入

```
做一页 自建链 技术定位与竞品对比。
对象：自建链 目标、自建链 现状、Tempo、Arc、Solana。
维度：定位、TPS、出块间隔、确认时间、手续费、节点数量、节点配置、共识算法、节点架构。
结论：自建链 目标是支付专用链，性能和成本目标接近新一代支付链，但现状与目标差距明显；Solana 可作为高性能参照但不是同类定位。
```

### 信息充足性判断

- 主题：自建链 技术定位与竞品对比 ✓
- 结论：目标定位成立，但现状与目标有差距，Solana 是参照对象 ✓
- 原始事实：5 个对象 × 9 个维度 ✓
- 关系：多对象、多维度对比 ✓

信息充足，但超过单页视觉预算，必须降维。

### Page Contract 示例

```
## Page Contract

### 1. 本页目标
面向 CTO，说明 自建链 的支付专用链定位、当前差距，以及与 Tempo / Arc / Solana 的关键差异。

### 2. 一句话结论
自建链 目标对齐支付专用链方向，但当前性能、节点配置和架构成熟度仍需从现状向目标收敛。

### 3. 使用的原始信息
- 对比对象：自建链 目标、自建链 现状、Tempo、Arc、Solana
- 对比维度：定位、TPS、出块间隔、确认时间、手续费、节点数量、节点配置、共识算法、节点架构
- 用户给出的结论：自建链 是支付专用链目标，Solana 作为高性能参照但定位不同

### 4. 页面内容范围
本页包含：
- 3 个 KPI cards：TPS、出块间隔、手续费/成本
- 重点矩阵：自建链 目标、自建链 现状、Tempo、Solana × 5 个关键维度
- 右侧 insight：自建链 目标与现状差距、Solana 仅作高性能参照
- 底部脚注区：Arc 和详细节点架构进入编号说明或建议拆页

本页不包含：
- 5 个对象 × 9 个维度的全量矩阵
- 每个节点架构的长段落解释
- 用户未提供的性能推断或路线建议

### 5. 推荐页面结构
- 主图：KPI cards + comparison-matrix 组合版式
- 表格：4 个对象 × 5 个关键维度，保留支撑结论的核心差异
- 文字：右侧 insight panel 不超过 3 条
- 结论区：Bottom bar 写"目标清晰，现状差距集中在性能、配置和架构成熟度"

### 6. 版面蓝图
- 主视觉区域：顶部 KPI cards 20%，中部重点矩阵 50%
- 证据/说明区域：右侧 20% 放 3 条 insight，不覆盖矩阵
- 脚注/来源区域：底部 8% 放编号说明，不能与表格重叠
- 需要压缩或不进入主画面的信息：Arc 与完整节点架构细节不进主表，建议拆成第二页全量附录

### 7. 生成前确认
请确认是否按这个内容范围生成当前页 PPT。
```

### 生成策略说明

- 不能直接生成 5 列 × 9 行大表。
- 先用 KPI cards 把 CTO 最关心的性能/成本指标前置。
- 主表只保留关键维度，避免单元格多行换行。
- 脚注必须有独立底部区域，不能覆盖表格。
- 如果用户要求保留所有对象和维度，建议拆成两页。
