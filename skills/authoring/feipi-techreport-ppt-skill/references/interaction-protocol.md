# 交互协议

## 交互目标

- **收敛**：每次交互都指向一个明确目标——确认 Page Contract、生成 PPT、或补充原始信息。
- **不问不必要问题**：默认假设已知，只问缺少的关键事实。
- **不让用户选风格**：风格、受众、颜色、密度全部内置默认值。
- **不把 skill 变成调研 agent**：不负责外部搜索、不负责补充未经用户提供的事实。
- **不把 skill 变成需求访谈机器人**：不引导用户做需求澄清，只做信息充足性检查。

## 默认交互流程

```
一页一交互
一页一确认
先检查信息
再生成决策确认卡片（内部完成完整 Page Contract 规划）
确认后才生成 PPT
```

每一页独立处理，不批量生成（除非用户明确要求多页规划）。

## 确认规则：决策确认卡片

信息充足后，**不要**把完整 Page Contract 原样展示给用户。用户可见输出改为"决策确认卡片"。

### 格式

```
信息足够，我建议按「[版式名称]」生成这一页。

需要你确认 [N] 个取舍：

1. [决策点名称]
   A. [选项描述]（推荐）
   B. [选项描述]
   C. [选项描述]（如有）

如无异议，我按 A + A 生成。
```

### 何时提问

只在**真正不确定、会影响产出**的地方提问：
- 对象是否压缩或拆页；
- 主画面优先级；
- 哪些信息进入主表、脚注或第二页；
- 结论聚焦方向存在多个合理选项时的取舍。

### 何时不提问

以下内容**由 skill 内部决策，不展示给用户**：
- 已确定的原始信息复述；
- 内部版面百分比分配；
- 颜色、字号、间距等默认视觉参数；
- 只有一个合理方案的版式选择。

### 无不确定性时

当没有明显不确定性时，只输出简短确认：
"信息足够，我将按「[版式名称]」生成，重点突出「[结论]」。确认后开始生成。"

### 选项约束

- 每个决策点提供 **2-3 个候选**，标注一个推荐项。
- 不要让用户开放式回答，而是让用户选择候选或接受默认组合。
- 用户可见确认控制在约 15 行以内。

## 追问规则

最多问 **3 个阻塞问题**。超过 3 个仍未补齐关键信息，停止追问并说明当前能做什么、不能做什么。

### 只问原始信息，不问偏好

**可以问：**

- 这一页想表达的核心结论是什么？
- 需要展示的对象/模块/阶段/方案有哪些？
- 每个对象有哪些事实、数据、特征或关系？
- 这些对象之间是流程、分层、对比、因果还是阶段关系？

**不要问：**

- 你喜欢什么风格？
- 你想给谁看？
- 要不要科技风？
- 要不要蓝色？
- 要不要高密度？
- 要不要加图标？

这些全部使用默认值。

## Fast Mode

如果用户明确说出以下任一表述：

- "不要确认，直接生成"
- "跳过 Page Contract"
- "你自己处理"
- "快速生成"

则可以进入 **fast mode**，跳过 Page Contract 确认环节，直接生成 PPT。

但即使 fast mode，也必须：

1. 先做信息充足性检查——信息不足时仍要要求补充。
2. 不做外部调研。
3. 不编造事实。
4. 不添加未经用户提供的数据、指标、结论。

Fast mode 只是跳过确认环节，不是跳过所有规则。

## 多页处理

如果用户要求一次处理多页：

1. 先输出多页规划（每页一句话主题）。
2. 逐页执行单页流程：信息检查 → Page Contract → 确认 → 生成。
3. 不要一次性生成全部 PPTX 而不做中间确认。

## 语言

默认使用中文回复。技术术语、缩写可以保留英文（CTO、PPTX、Page Contract、Visual QA、Architecture Map 等）。
